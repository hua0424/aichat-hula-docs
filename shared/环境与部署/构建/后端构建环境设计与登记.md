# 后端构建环境设计与登记（AIChat / HuLa-Server + aichat-plugins）

## 结论
后端开发采用**个人开发目录 + Docker 容器化构建/运行**模式，并按职责拆分：

- HuLa-Server：开发容器与构建/运行容器分离（仅运行 4 个核心微服务，`luohuo-system-server` **不使用** — 该服务为 AI/LLM 能力层，项目不使用 Hula 的 LLM 对话功能）
- aichat-plugins：开发与运行复用同一容器
- 第三方基础服务仅新建 Nacos；MySQL/Redis/RocketMQ 连接外部现有服务

## 二、镜像与容器规划

### 1) HuLa-Server（分离）
1. `hula-server-dev`
   - Dockerfile: `infra/docker/hula-server/Dockerfile.dev`
   - 基础镜像: `maven:3.9-eclipse-temurin-21`
   - 用途: 本地开发、调试、Maven 测试、源码挂载

2. `hula-server-runtime`
   - Dockerfile: `infra/docker/hula-server/Dockerfile.runtime`
   - 基础镜像: `eclipse-temurin:21-jre`
   - 用途: 运行环境，通过 `start-runtime.sh` 从 workspace bind-mount 复制 JAR 并启动全部服务。更新方式：`docker restart hula-server-runtime`（自动取最新编译产物）

### 2) aichat-plugins（开发+运行）
1. `aichat-plugins-dev`
   - Dockerfile: `infra/docker/aichat-plugins/Dockerfile.dev`
   - 基础镜像: `node:24-bookworm`
   - 用途: 开发与运行共用容器（源码挂载 + sleep infinity）
   - 内置 openclaw@latest（AI 引擎网关），启动时 symlink aichat-claw plugin 并运行 openclaw gateway

### 3) 第三方服务
1. `nacos`
   - 镜像: `nacos/nacos-server:v2.3.2`
   - 用途: 本地新增唯一第三方基础服务

### 端口来源
- `infra/env/.env.common`：默认端口定义
- `infra/compose/docker-compose.dev.yml`：端口映射关系（host:container）

### 联调变更约定（强制）
- 任何端口改动必须同步更新本页（提交同一 PR）。
- 端口被占用时，优先改宿主机端口（保留容器内端口不变），并同步更新前端 `.env` 与本页。
- 对外通知格式：`服务名 + 新端口 + 生效时间 + 影响范围`。

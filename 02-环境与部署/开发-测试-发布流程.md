# 开发-测试-发布流程（Docker化）

## 结论
采用“**开发提交代码 + CI构建镜像 + 运维手动审批发布测试环境 + QA验收**”流程。测试机不直接拉源码构建。

## 适用范围
- HuLa（客户端）
- HuLa-Admin（管理端）
- HuLa-Server（服务端）

## 流程总览
1. 开发在 `feature/*` 分支开发（本地代码 + dev容器）
2. 本地容器内完成编译与单元测试
3. 提交 PR 合并到 `dev`
4. CI 构建镜像并打测试标签（如 `dev-<shortsha>`）
5. 运维在测试环境 `docker-compose` 更新镜像 tag，**手动审批**后发布
6. QA 在测试环境执行回归/验收
7. 通过后进入主线发布流程（`master`）

## 职责分工
- 开发：代码实现、单测、PR、变更说明
- 运维：Compose编排、环境配置、发布与回滚
- QA：测试用例执行、缺陷闭环、验收结论
- Manager：节点评审、决策与风险仲裁

## 发布原则
- 测试环境只消费镜像，不消费源码
- 每次发布必须可追溯到 commit 与 PR
- 发布必须走人工审批
- 回滚优先使用上一个稳定镜像 tag

## 镜像仓库策略（当前）
- 当前阶段：先使用**本地镜像存储**（单机/局域环境）
- 后续阶段：切换到集中镜像仓库（Harbor/其他）

## 测试环境发布步骤（运维）
1. 获取待发布镜像 tag（来自 CI 产物）
2. 修改 `docker-compose.yml` 对应服务镜像版本
3. 审批通过后执行：
   - `docker compose pull`（或本地已存在可跳过）
   - `docker compose up -d`
4. 观察健康检查与核心日志
5. 通知 QA 开始验收

## 回滚步骤
1. 确认上一稳定 tag
2. 回改 compose 镜像版本
3. `docker compose up -d`
4. 复测核心链路

## 质量门禁（建议）
- 构建成功
- 单元测试通过
- 关键静态检查通过
- 镜像可启动且健康检查通过

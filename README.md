# hula-docs

AIChat（基于 HuLa）项目统一文档仓库。

## 目标
- 让新成员 **30 分钟内**了解项目全貌（目录、技术栈、环境、启动方式）
- 统一管理：需求、设计、任务、进度、问题、决策、发布
- 降低信息分散导致的沟通与交付风险

## 目录导航

| 区域 | 目录 | 用途 |
|------|------|------|
| 看板 | `看板.md` | 全局任务看板（TODO/DOING/BLOCKED/DONE） |
| 共享知识 | `shared/` | 长期维护的参考文档 |
| 活跃需求 | `active/` | 进行中的 REQ，一个需求一个目录 |
| 归档 | `archive/` | 已完成的需求，历史参考 |
| 模板 | `.templates/` | 新建文档时的骨架模板 |

### shared/ 共享知识

| 目录 | 内容 |
|------|------|
| `shared/项目总览.md` | 交付边界、里程碑、代码仓结构 |
| `shared/开发规范/` | Git 分支/提测/提交规范 |
| `shared/环境与部署/` | 联调地址、构建手册、CI 流程、依赖拓扑、镜像基线 |
| `shared/决策记录/` | 项目级 ADR（流程/架构原则） |
| `shared/运维与发布/` | 发布清单、回滚预案 |
| `shared/风险与问题/` | 全局风险台账、阻塞问题追踪 |

### 快速入口
- 项目总览：`shared/项目总览.md`
- 联调环境：`shared/环境与部署/联调/联调环境说明.md`
- 构建手册：`shared/环境与部署/构建/后端CI构建手册.md`
- 看板：`看板.md`

## 使用原则
1. **结论先行**：每个文档开头写「结论/状态」
2. **单一事实源**：同类信息只放一个地方，其他文档链接引用
3. **可追溯**：重大结论必须在 `shared/决策记录/` 留痕
4. **可执行**：任务必须有 owner、输入输出、截止时间

## REQ 生命周期

```
新建 REQ → active/REQ-XXX/
              需求.md + 设计-*.md + 任务-*.md + adr/

完成后  → 1. 提取可复用知识 → 回写 shared/ 对应位置
          2. 整个目录移入 archive/
```

## 协作节奏
- 每日：更新 `看板.md`
- 每次评审后：更新 REQ 目录下的设计/任务文档 + 必要时写 ADR
- 每次发布前后：更新 `shared/运维与发布/发布清单.md`
